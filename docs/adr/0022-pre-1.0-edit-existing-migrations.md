# 0022 — Pre-1.0, edit existing migrations in place

- **Status:** Accepted, then **amended 2026-06-27** — the in-place rule applied through **0.9.4 only**; from **0.9.5** on, schema changes are append-only altering migrations (pre-release deployments now exist). **Further amended 2026-06-28** — a one-time pre-1.0 baseline reset folded the 0.9.6 cross-migration ALTERs back into their CREATEs and renumbered to a gapless sequence; the bright line is restated as "no in-place once you can't wipe," which still binds from 1.0. See the Amendment sections below.
- **Enforced by:** convention + review. Through 0.9.4 a schema issue was fixed in the original migration file (zero shipped installations to migrate); the `build.rs` `rerun-if-changed` on the migrations dir ensures such an in-place edit recompiles the embedded schema. From 0.9.5 on, a schema change is a new numbered `ALTER` migration instead.
- **Supersedes:** —

## Amendment (2026-06-27) — in-place edits end after 0.9.4

The decision below rested on "pre-1.0 there are no shipped installations." That
premise held through **0.9.4**. As of **0.9.5** it no longer holds: pre-release
deployments now run with the schema applied (the public dogfood and the internal
platform mirror), and we intend to keep them. Editing a migration in place
changes its checksum, which `sqlx::migrate!` rejects against a database that
already applied the old version (`VersionMismatch`) — so an in-place edit now
breaks every live pre-release setup, not merely re-shapes a fresh install.

**Revised rule (effective 0.9.5+):** schema changes are **append-only altering
migrations** — add a new numbered migration file with `ALTER TABLE …`; never
edit an existing migration file. This is the post-1.0 append-only rule, pulled
forward to 0.9.5 because deployments now exist. The in-place guidance below
still describes how the migrations that predate 0.9.5 (≤ 0.9.4) were maintained;
those files stay as they are.

## Amendment (2026-06-28) — one-time pre-1.0 baseline reset

The 0.9.5 amendment above pulled the append-only rule forward on the premise
"deployments now exist." That premise is correct but the 0.9.5/0.9.6 deployments
are **wipeable dogfood/test databases** (the public dogfood `registry.hort.rs`
and the internal platform mirror `hort.kdp`), not a non-wipeable production
store. The append-only-from-0.9.5 rule therefore over-tightened: it accumulated a
three-link forward-ALTER chain (the 0.9.6 work) on a schema whose every live
copy can be — and, in this reset, is — dropped and recreated, paying the
`CREATE`-then-`ALTER` archaeology cost for zero compatibility benefit.

**This one-time reset** folds the three 0.9.6 cross-migration ALTERs back into
their defining CREATEs and renumbers the chain to a gapless sequence:

- `negligible_action` (formerly an `ALTER TABLE policy_projections ADD COLUMN`)
  is now a column of `CREATE TABLE policy_projections` in `005_policy.sql`.
- `informational_class` (formerly an `ALTER TABLE scan_findings ADD COLUMN`)
  is now a column of `CREATE TABLE scan_findings` in
  `009_scan_jobs_and_findings.sql`.
- the `'policy-reevaluation'` worker task kind (formerly a
  `DROP CONSTRAINT jobs_kind_check; ADD CONSTRAINT …` widening) is now a member
  of the inline `jobs.kind` CHECK IN-list in
  `009_scan_jobs_and_findings.sql` — no drop+re-add.

The chain is renumbered to gapless `001`–`012` (the prior `011 → 013` gap is
closed by `013_subscriptions.sql → 012_subscriptions.sql`). Applying the reset
requires wiping the dogfood databases — acceptable precisely because **no
non-wipeable production database exists yet**.

**The bright line (re-affirmed and made the controlling principle):** in-place
edits and renumbering are forbidden the moment a non-wipeable production
database exists — which is **from 1.0**. At that point
`ALTER`-as-a-new-numbered-migration resumes and is permanent. The rule is **"no
in-place once you can't wipe,"** not "never in-place": the 0.9.5 amendment's
"append-only from 0.9.5" wording was the operational stand-in for that
principle while the wipeable dogfood deployments existed; this reset corrects
it to the principle itself. A single reset that lands a clean canonical baseline
is the right pre-1.0 endpoint; the squash-at-1.0 alternative below is its
post-1.0 mirror.

## Context

After v1.0 ships, migrations are an append-only chain — every deployed instance has applied migration N, so fixing N means adding N+1. But **pre-1.0 there are no shipped installations**: appending a corrective `ALTER` on top of a migration that has never run anywhere creates a permanent two-step (`CREATE` then immediately `ALTER`) that every future fresh install pays forever, for no compatibility benefit.

## Decision

Until the v1.0 schema freeze, fix a schema problem **by editing the original migration in place**, not by appending an ALTER chain. This consolidates what would otherwise be permanent corrective migrations into the single canonical migration (the prototype's `082/083/086–089/092–094`-style standalone numbers were consolidated this way). The embedded-schema staleness hazard is handled structurally: each consuming crate's `build.rs` points `cargo:rerun-if-changed` at the migrations directory, so an in-place edit recompiles the `sqlx::migrate!`-embedded schema rather than leaving a stale cached copy.

## Consequences

- Fresh installs get a clean, minimal migration chain rather than `CREATE`-then-`ALTER` archaeology.
- This decision **expires at v1.0**: once installations exist, migrations become strictly append-only and this ADR no longer applies.
- In-place edits are safe only because `rerun-if-changed` forces recompilation; without it a warm `target/` cache would embed a stale schema (the failure mode the build scripts exist to prevent).

## Alternatives considered

- **Append-only migrations from day one.** Rejected pre-1.0: pays a permanent N-files-forever cost for a compatibility property that has no consumers yet (no shipped installs).
- **Squash migrations at v1.0 instead of editing in place.** A valid endpoint, but editing in place during development keeps the chain clean continuously rather than deferring a big squash; either way the chain is append-only after 1.0.
