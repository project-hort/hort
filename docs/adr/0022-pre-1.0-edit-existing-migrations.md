# 0022 — Pre-1.0, edit existing migrations in place

- **Status:** Accepted (time-bounded: applies until the v1.0 schema freeze)
- **Enforced by:** convention + review — until v1.0 there are zero shipped installations to migrate, so a schema issue is fixed in the original migration file, not by appending an `ALTER` migration on top. The `build.rs` `rerun-if-changed` on the migrations dir ensures an in-place edit recompiles the embedded schema.
- **Supersedes:** —

## Context

After v1.0 ships, migrations are an append-only chain — every deployed instance has applied migration N, so fixing N means adding N+1. But **pre-1.0 there are no shipped installations**: appending a corrective `ALTER` on top of a migration that has never run anywhere creates a permanent two-step (`CREATE` then immediately `ALTER`) that every future fresh install pays forever, for no compatibility benefit.

## Decision

Until the v1.0 schema freeze, fix a schema problem **by editing the original migration in place**, not by appending an ALTER chain. This consolidates what would otherwise be permanent corrective migrations into the single canonical migration (the prototype's `082/083/086–089/092–094`-style standalone numbers were consolidated this way). The embedded-schema staleness hazard is handled structurally: each consuming crate's `build.rs` points `cargo:rerun-if-changed` at the migrations directory, so an in-place edit recompiles the `sqlx::migrate!`-embedded schema rather than leaving a stale cached copy.

## Consequences

- Fresh installs get a clean, minimal migration chain rather than `CREATE`-then-`ALTER` archaeology.
- This decision **expires at v1.0**: once installations exist, migrations become strictly append-only and this ADR no longer applies.
- In-place edits are safe only because `rerun-if-changed` forces recompilation; without it a warm `target/` cache would embed a stale schema (the failure mode the build scripts exist to prevent).

## Alternatives considered

- **Append-only migrations from day one.** Rejected pre-1.0: pays a permanent N-files-forever cost for a compatibility property that has no consumers yet (no shipped installs).
- **Squash migrations at v1.0 instead of editing in place.** A valid endpoint, but editing in place during development keeps the chain clean continuously rather than deferring a big squash; either way the chain is append-only after 1.0.
