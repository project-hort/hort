# ISO/IEC 27001:2022 — operating Hort inside an ISMS

This document is for an operator who runs Hort as a technical component inside an
organisation's **Information Security Management System (ISMS)** and needs to show
an ISO/IEC 27001:2022 auditor how the deployment supports the relevant Annex A
controls. It is the security-controls counterpart to the data-protection record in
[`docs/compliance/GDPR.md`](GDPR.md): where that document is the legal artefact for
event-log retention and erasure, this one is the controls-traceability artefact for
the broader information-security regime.

The engineering substance behind every claim here lives in the architecture
explainers — [`security.md`](../architecture/explanation/security.md),
[`event-sourcing.md`](../architecture/explanation/event-sourcing.md),
[`cas-storage.md`](../architecture/explanation/cas-storage.md),
[`scanning-pipeline.md`](../architecture/explanation/scanning-pipeline.md) — the
inbound-auth control spec in [`docs/auth-catalog.md`](../auth-catalog.md), and the
standing decisions in [`docs/adr/`](../adr/). This document maps those facts onto the
Annex A control framework. Where a control is purely organisational, this document
says so rather than inventing a technical hook for it.

> **Hort is not "ISO 27001 certified", and this document is not a certificate or a
> conformity assessment.** ISO 27001 certifies an *organisation's ISMS*, never a
> single piece of software. Hort is a tool that, configured as described below,
> *supports* an operator's implementation of specific Annex A controls. The
> auth-catalog states the same boundary for its own scope
> ([`docs/auth-catalog.md`](../auth-catalog.md) §1.1): a control inventory is not an
> attestation. Treat the mappings here as "here is the lever Hort gives you for this
> control", not "this control is satisfied because you deployed Hort".

## 1. Scope and the shared-responsibility model

The boundary of an ISO 27001 certification is the ISMS — its people, processes,
governance, risk treatment, and the assets in scope. A Hort deployment is one asset
inside that boundary. Certification covers the management system that surrounds the
tool; it never covers the tool in isolation.

Every control area below splits the same way, and the split recurs throughout this
document:

| | **Hort provides (the platform-side lever)** | **The operator operates (the ISMS-side obligation)** |
|---|---|---|
| Identity | OIDC validation, claim-based RBAC, native-token kinds, short-lived sessions | The IdP itself, MFA, joiner/mover/leaver process, access reviews |
| Cryptography | SHA-256 CAS, Argon2id token hashing, Ed25519 token signing, TLS-builder posture | Key custody, rotation cadence, the CA bundle's integrity and RBAC |
| Audit | Append-only event store, tamper-evident chain, structured tracing | Log shipping, SIEM, retention enforcement runner, alerting, review |
| Vulnerability mgmt | Scan pipeline, quarantine gate, advisory watch, provenance verify | Triage process, exception governance, patch SLAs, risk acceptance |
| Supply chain | Mandatory upstream checksum verify, provenance gates, CAS immutability | Trust-root provisioning, allowlist policy, publisher-identity policy |
| Config & change | Gitops config tree, strict Helm schema, apply-time linter | Git review workflow, segregation of duties, change approval |

The recurring rule: **Hort gives you a control surface that is fail-closed and
auditable by construction; the ISMS process that drives it is yours.** Hort cannot
run your access reviews, set your risk appetite, or own your key custody — but it can
make those activities evidence-producing and hard to bypass.

### What is structurally outside Hort's scope

These belong to the ISMS and have no Hort-side lever at all (see §5 for the honest
list): physical and environmental security (A.7), human-resources security (A.6),
the ISMS governance clauses (Clauses 4–10 of the main standard, including risk
assessment and management review), supplier-relationship governance beyond the
technical proxy controls (A.5.19–A.5.22), and any SIEM/log-analytics platform —
Hort emits records, it does not analyse them.

## 3. Operational procedures

Runbook-style guidance for standing up and operating Hort in a way that produces
the evidence an ISO 27001 auditor expects. This complements, and does not replace,
the [security-hardening-checklist](../architecture/how-to/deploy/security-hardening-checklist.md).

### 3.1 Deployment hardening (A.8.9, A.8.20, A.8.22)

- **TLS everywhere, no escape hatch.** Hort speaks plain HTTP and expects a
  TLS-terminating reverse proxy in front; set `HORT_PUBLIC_BASE_URL` to the
  `https://` URL (this also drives HSTS emission at the proxy and the OCI
  `WWW-Authenticate` realm). Set `HORT_REQUIRE_HTTPS=true` (Helm default) so the
  binary refuses to boot without positive TLS evidence. There is **no
  `*_INSECURE_TLS` knob** anywhere (ADR 0010) — internal CAs are trusted only via
  `HORT_EXTRA_CA_BUNDLE`.
- **Least-privilege DB split (ADR 0009).** Provision three Postgres roles
  ([postgres-roles.md](../architecture/how-to/deploy/postgres-roles.md)):
  `hort_admin` (DDL, runs `hort-server migrate` as a separate Job, wired via
  `postgres.admin.existingSecret`), `hort_app_role` (DML-only runtime, wired via
  `postgres.app.existingSecret`), and the migration-created `NOLOGIN`
  `hort_retention_role` (the only role that may `DELETE` from `events`, via the
  immutability trigger's `SET LOCAL ROLE`). The runtime never holds DDL; the
  migrate DSN is never mounted on the serving Deployment. `PgEventStore::new` probes
  the privilege at boot and refuses to start if the runtime role can mutate `events`.
- **Network segmentation (A.8.22 / control-plane tiers).** Move admin and
  subscription routes onto an internal-only listener with `HORT_CONTROL_BIND`
  (binding an unspecified address additionally requires `HORT_CONTROL_PUBLIC_BIND`);
  keep `networkPolicy.enabled=true` (chart default) so the artifact plane (`:8080`)
  and the control plane (`:9443`) are separable, and the egress policy bounds
  webhook SSRF blast radius. The token-generation plane is **public by requirement**
  with no network backstop — its application-layer controls (anti-replay, audience
  binding, rate limits, short TTLs) are load-bearing.
- **Pod hardening.** The chart defaults to a non-root UID, `readOnlyRootFilesystem`,
  dropped capabilities, `seccompProfile: RuntimeDefault`, and
  `allowPrivilegeEscalation: false` — PSS-restricted-aligned. Keep these defaults.

### 3.2 Identity & access management + access reviews (A.5.15–18, A.8.2–3)

- Drive all RBAC through `$HORT_CONFIG_DIR`: `ClaimMapping` (IdP group → claim) and
  `PermissionGrant` (`claims`- or `user`-subject, with `read`/`write`/`delete`/
  `admin`/`curate`/`admin_task_invoke`/`prefetch` permissions, optionally repo-scoped).
- **Periodic access review** is a `git` exercise: the gitops tree *is* the
  authoritative access matrix, so a review is a diff over the committed objects plus
  the apply-linter output. The admin endpoint
  `GET /api/v1/admin/users/:user_id/effective-permissions` resolves what a given principal
  can actually do — use it to spot-check the review.
- Keep admin authority narrow: `HORT_TOKEN_ALLOW_ADMIN=false` by default; admin
  capability on a CLI session is hard-capped at ≤15 min; service accounts cannot be
  admin (rejected at apply).
- For CI/CD, prefer **OIDC federation** over long-lived PATs: declare an `OidcIssuer`
  + a `ServiceAccount` with non-empty, discriminating `federatedIdentities[].claims`
  (an empty `claims` map is a hard apply-time reject — it means "any JWT from this
  issuer"). See [federate-ci-oidc.md](../architecture/how-to/federate-ci-oidc.md).

### 3.3 Cryptography, key & secret management (A.8.24, A.5.17)

- **Secrets are operator-wired, never in the tree.** Hort embeds no Vault/KMS SDK;
  reference secrets via `secretRef` (`source: file | env_var`) and contain the file
  adapter with `HORT_SECRETS_FILE_ROOT`. `source: file` re-reads on each resolve, so
  rotation needs no restart; `source: env_var` is immutable until restart. See
  [wire-secrets.md](../architecture/how-to/wire-secrets.md).
- **The CA bundle is an auth-critical asset.** `HORT_EXTRA_CA_BUNDLE` is additive
  across *all* outbound TLS surfaces including OIDC/JWKS, so a compromised bundle can
  impersonate the IdP. Source it from an RBAC-restricted `ClusterTrustBundle` (not a
  namespace-editable ConfigMap), at least as tightly controlled as the OIDC client
  secret, and prefer a name-constrained intermediate. Set-but-unreadable is
  fail-closed (boot abort); `hort_extra_ca_anchors` reports the loaded count.
- **Token-signing-key rotation.** The Ed25519 OCI/CLI-session signing key
  (`HORT_OCI_TOKEN_SIGNING_KEY[_FILE]`) rotates with a verify-only previous key
  (`_PREV[_FILE]`); keep the overlap window ≥ the CLI-session TTL. See
  [rotating-service-account-tokens.md](../architecture/how-to/rotating-service-account-tokens.md)
  for automated PAT rotation (`scheduledTasks.serviceAccountRotation`), which enforces
  `validity ≥ 2 × rotationInterval` so consumers always have an overlap grace window.
- **Key custody is the operator's.** Hort consumes keys; it does not generate,
  escrow, or custody them.

### 3.4 Logging & monitoring (A.8.15–16)

- **What the event store gives you:** an append-only, tamper-evident record of every
  artifact-lifecycle, policy, authorization, and security-relevant administrative
  action, immutable at the DB level (separate-role trigger) and refusing to boot if
  that trigger is missing. Schedule `verifyEventChain` (`scheduledTasks.verifyEventChain`)
  and alarm on the `hort_event_chain_verify_overdue` gauge.
- **What tracing gives you:** the *attempted-but-not-recorded* surface (auth
  failures, authz denials, rate-limit rejects) as single-line structured JSON, sized
  for fail2ban/SIEM consumers. Privilege denials log at `info!`, not `error!`.
- **What metrics give you:** the catalogued `hort_*` Prometheus surface for trend and
  alerting. Keep `HORT_METRICS_REQUIRE_AUTH=true`; bind metrics to loopback or a
  dedicated listener (`HORT_METRICS_BIND`), and require `HORT_METRICS_PUBLIC_BIND` to
  expose it. Cardinality is bounded by design — no `user_id`, `client_ip`,
  `artifact_id`, or path/version strings appear as labels (those live in the event
  store and tracing).
- **Retention.** Event-log retention floors and the GDPR erasure-exemption rationale
  are in [`GDPR.md`](GDPR.md); automated retention can only target the four permitted
  stream categories (`Artifact`, `AuthAttempts`, `DownloadAudit`, `TokenUse`), a
  guard-tested invariant (ADR 0030). Schedule the retention tasks
  (`scheduledTasks.retentionEvaluate` / `retentionPurge`) per your documented floors.

### 3.5 Vulnerability & patch management (A.8.8, A.5.7)

- Enable both scanner backends (`HORT_SCANNER_TRIVY_ENABLED=true`,
  `HORT_SCANNER_OSV_ENABLED=true`); the worker refuses to boot with both disabled.
- Define the gate in `ScanPolicy` (severity threshold, license policy, quarantine
  duration, `scanBackends`). An ingested artifact is **quarantined and
  non-downloadable** until released, and release is fail-closed: a clean scan alone
  does not release — only the timer with a recognised authority, an admin override,
  a curator waiver, or policy re-evaluation does. A scan that exhausts its retry
  budget becomes terminal `scan_indeterminate` (never timer-released).
- Keep verdicts fresh: schedule `cronRescanTick` (interval-driven safety net) and
  `advisoryWatchTick` (targeted re-scan on new disclosures). The `cron` path is
  independent of the advisory feed, so a silent feed does not silence re-adjudication.
- Patch the *platform's own* dependency graph through the project's CI security gates
  (`cargo audit --deny warnings`, `cargo deny check`) — these run against the live
  RustSec advisory DB.

### 3.6 Software-supply-chain integrity (A.5.23, A.8.28)

- **Mandatory upstream verification (ADR 0006):** every pull-through fetch verifies a
  protocol-native digest (OCI manifest digest, npm `dist.integrity`, PyPI
  `digests.sha256`, Cargo `cksum`) before bytes are stored. There is
  no opt-out; a tampered upstream response is rejected at the boundary.
- **Provenance verification (ADR 0027):** enable cosign verification on the worker
  (`HORT_PROVENANCE_COSIGN_ENABLED=true`) with a pinned, operator-provisioned
  `HORT_PROVENANCE_TRUSTED_ROOT_FILE` (verification is fully offline — no live
  Rekor/Fulcio fetch, no SSRF surface). Per repo, set `ScanPolicy.provenanceMode`
  (`off` / `verify_if_present` / `required`); `required` is a fail-closed
  AND-precondition on release (a `Pending` provenance clearance blocks the timer) and
  rejects at apply if `provenanceIdentities` is empty. See
  [enable-provenance-verification.md](../architecture/how-to/enable-provenance-verification.md).
- **CAS immutability (ADR 0003):** stored bytes are addressed by their own SHA-256
  and cannot be silently replaced; read-time verification (`VerifyingReader`) and the
  background `hort-server scrub` sweep
  (`HORT_CAS_SCRUB_ACTION_ON_MISMATCH=alert|tombstone`) detect at-rest corruption.
- For downstream verification of Hort's *own* release binaries and SBOM (Sigstore
  keyless), see [release-verification.md](../architecture/how-to/release-verification.md).

### 3.7 Backup, recovery & resilience (A.8.13–14)

Hort has no built-in backup subsystem — this is an operator responsibility on the two
state stores it depends on:

- **Postgres** holds the event store (source of truth), projections, RBAC, and config
  state. Back it up with your standard DBA process; the event store is append-only, so
  point-in-time recovery is well-behaved.
- **CAS backend** (filesystem / S3 / GCS / Azure) holds the immutable artifact bytes.
  Because content is addressed by SHA-256, a restored blob is self-verifying; run
  `hort-server scrub` after a restore to confirm integrity.
- A restored deployment re-applies `$HORT_CONFIG_DIR` at boot, so config recovery is
  the git checkout. Capacity planning must account for orphan growth (failed ingests
  leave addressable-but-unreferenced blobs; reference-counted GC is deferred).

### 3.8 Incident response & change management (A.5.24–28, A.8.32)

- **IR evidence** is the event store plus tracing: `AuthenticationAttempted` events
  carry `client_ip` for the incident-investigation horizon; authz denials and
  rate-limit rejects land in structured tracing for SIEM correlation.
- **Change management** is config-as-code: every RBAC, policy, and machine-identity
  change is a reviewed git commit that emits a durable authorization event on apply.
  Destructive tasks are idempotent per UTC day, so a cron restart cannot double-fire a
  retention purge.
- Hort does not page, ticket, or run your IR playbook — it produces the records those
  processes consume.

| **Effective-permission snapshots** | `GET /api/v1/admin/users/:user_id/effective-permissions` for any principal | A.5.15, A.8.2 |
| **RBAC / policy config-as-code** | The committed `$HORT_CONFIG_DIR` tree + apply-linter output (`hort_apply_config_linter_total`) | A.5.15, A.8.9 |
| **Operational telemetry** | The `hort_*` metrics surface (auth results, authz decisions, rate-limit, CAS-integrity failures, advisory-feed volume, event-chain-overdue) | A.8.16 |
| **Privilege-separation proof** | Postgres role grants + the boot-time privilege probe that fails closed if the runtime role can mutate `events` | A.8.2, A.8.15 |
| **Crypto posture** | Absence of any `*_INSECURE_TLS` knob; `hort_extra_ca_anchors` gauge; Argon2id hashing (bcrypt structurally banned, guard-tested) | A.8.24 |
| **Admin-action attribution** | `ArtifactReleased` carries `released_by_user_id` + a mandatory ≤512-byte `justification` (required for admin/curator releases); `AdminStatusChanged` records every admin-bit flip | A.8.2, A.5.28 |

Auditor-friendly properties worth stating: the event store is the *system of record*
(artifact state is a projection of it, not the other way around), every gitops apply
attributes its events to `Actor::GitOps`, and `Actor::Internal` is unforgeable from
any inbound-HTTP path — so an event's provenance inside the log is trustworthy.

## 5. Limitations and operator responsibilities

Stated plainly, so no one mistakes a tool for an ISMS. Hort does **not** provide:

- **The ISMS itself.** Risk assessment and treatment, the Statement of Applicability,
  management review, internal audit, continual improvement, and the policy set
  (Clauses 4–10; A.5.1) are organisational and entirely out of scope. A Hort
  deployment is in-scope *as an asset*; it is not the management system.
- **Physical & environmental, and people, controls.** A.7.* and A.6.* have no Hort
  lever.
- **TLS termination, HSTS, and network-layer DDoS defence.** Hort speaks plain HTTP
  behind a reverse proxy; its per-IP rate limiting absorbs small-scale abuse, not a
  volumetric attack.
- **A SIEM or log-analytics platform.** Hort *emits* tamper-evident events and
  structured traces; analysis, correlation, alerting, and long-term archival are the
  operator's pipeline.
- **Key custody, generation, or escrow.** Hort consumes operator-supplied keys and
  CA bundles; it generates none, custodies none, and embeds no Vault/KMS client.
- **Retention enforcement as a runtime feature.** The floors and legal basis are
  documented ([`GDPR.md`](GDPR.md)); the deletion is performed by operator-scheduled
  tasks under a dedicated DB role.
- **General-purpose anti-malware.** The scanning pipeline adjudicates
  dependency/CVE/license findings against the actual bytes; it is not a heuristic AV
  engine for arbitrary content and should not be presented as satisfying A.8.7's
  malware-control process wholesale.
- **Multi-tenant workload isolation.** Repositories are scoped and authorized, but
  workload isolation between tenants on a shared deployment is out of scope today.
- **Secret distribution.** Secrets are operator-wired via `secretRef`; the
  distribution trust boundary (ESO, Vault, CSI, k8s Secrets) is external and audited
  independently.

The single recurring obligation behind all of these: **Hort makes the technical
controls fail-closed and evidence-producing; the operator owns the management system
that decides, reviews, and governs them.** That division is the whole point of the
shared-responsibility model in §1.
